# Design: BUG-3 Phase Token Redesign

**Date:** 2026-03-25
**ADR:** [2026-03-25-phase-token-security-model.md](2026-03-25-phase-token-security-model.md)
**Security Model:** [2026-03-25-phase-transition-security.md](2026-03-25-phase-transition-security.md)

## Problem

The phase token system prevents Claude from self-triggering phase transitions by requiring one-time tokens generated by UserPromptSubmit (user-only hook). Five flaws make the current implementation insecure or brittle:

1. Token forgery during IMPLEMENT/REVIEW (bash-write-guard allows all writes)
2. Bare command matching generates false-positive tokens
3. 60-second TTL breaks long pipelines and has a future-timestamp bypass
4. Pre-generated "off" token for `/complete` conflates user and automated actions
5. OFF transition is not user-initiated

See the ADR for full research into platform constraints and approach selection rationale.

## Chosen Approach

**Approach B: Simplified File-Token with Application-Level Enforcement.** No HMAC — the security rests on the platform guarantee that UserPromptSubmit only fires on user input. Defense-in-depth via PreToolUse guards (unreliable but raises the bar). Same model as [cc-sessions](https://github.com/GWUDCAP/cc-sessions).

## Design

### Section 1: Token Generation (`plugin/scripts/user-phase-token.sh`)

**Slash command matching (explicit only):**
- `/define`, `/discuss`, `/implement`, `/review`, `/complete`, `/off` → phase token
- `/autonomy <level>` → autonomy token
- No bare `set_phase` or `set_autonomy_level` matching
- Regex: `^\s*/<command>(\s|$)` — must start with `/`, anchored to line start

**Token format (no timestamp):**
```json
{"target": "<phase_or_autonomy:level>", "nonce": "<random_hex_32>"}
```

**Token lifecycle:**
- One token per matching slash command
- Written to `$CLAUDE_PLUGIN_DATA/.phase-tokens/<nonce>`
- No TTL, no expiry, no cleanup loop
- Consumed immediately when `set_phase()` or `set_autonomy_level()` runs
- `/complete` generates only a "complete" token (no pre-generated "off")

**Added:**
- `/off` matching in slash command regex chain (new command — generates "off" target token)

**Removed:**
- Lines 34-36: bare `set_phase` matching
- Lines 48-54: bare `set_autonomy_level` matching
- Lines 63-71: expired token cleanup loop
- Lines 82-87: pre-generated "off" token for `/complete`
- Timestamp field from token JSON

### Section 2: Token Verification (`plugin/scripts/workflow-state.sh`)

**`_check_phase_token()` simplified:**
```
For each file in $token_dir/*:
  1. Atomic consume: mv $token_file → $token_dir/.consumed.$$
  2. Read target from consumed file
  3. Delete consumed file
  4. If target matches requested phase → return 0 (authorized)
Return 1 (no matching token found)
```

No timestamp check. No TTL. No future-timestamp guard. Just target matching with atomic consumption.

**`set_phase()` authorization logic (unchanged structure):**
1. Check 1: `_check_phase_token "$new_phase"` — valid one-time token
2. Check 2: Forward-only auto-transition — autonomy=auto, new ordinal > current ordinal, not OFF
3. If neither: BLOCKED with stderr message

**`set_autonomy_level()` unchanged** — delegates to `_check_autonomy_token()` which calls `_check_phase_token "autonomy:$level"`.

**`WF_SKIP_AUTH=1` bypass preserved** for test infrastructure.

### Section 3: Defense-in-Depth — Token Directory Guard (`plugin/scripts/bash-write-guard.sh`)

**New guard added BEFORE the implement/review early-exit:**

```bash
# Block writes to .phase-tokens directory in ALL phases (defense-in-depth)
if echo "$COMMAND" | grep -qE '\.phase-tokens'; then
    emit_deny "BLOCKED: Direct writes to the phase token directory are not allowed. Phase tokens are generated by the workflow system."
    exit 0
fi
```

**Placement in code flow:**
```
Parse COMMAND → Allow workflow-state.sh source → Allow git commit
→ Strip safe redirects → Detect python writes
→ NEW: Block .phase-tokens access (all phases, before implement/review exit)
→ Autonomy "off" check → implement/review exit 0
→ Phase-specific whitelist checks → Deny
```

**Why before implement/review exit:** The current `implement|review) exit 0` at line 122-124 allows all writes. The token directory guard must fire before this early return to catch forgery attempts during those phases.

**Honest caveat:** PreToolUse blocking is unreliable (see ADR). This guard is defense-in-depth, not a security boundary.

### Section 4: New `/off` Command (`plugin/commands/off.md`)

**New file.** User command to close the workflow.

**Template content:**
```
Transition the workflow to OFF phase.

<bash>
WF="${CLAUDE_PLUGIN_ROOT}/scripts/workflow-cmd.sh" && "$WF" set_phase "off" && "$WF" set_active_skill "" && echo "Phase set to OFF — workflow enforcement disabled."
</bash>

Confirm to the user that the workflow is closed.
```

**Impact on `/complete` (Step 9):**

Current Step 9 calls `set_phase "off"` directly. Change to:

```
Workflow complete. Run `/off` to close the workflow.
```

This ensures OFF is always user-initiated, consistent with the security model.

**Hard gate note:** `/off` calls `set_phase "off"`, which enforces the same hard gates as any phase transition. From COMPLETE phase, all completion pipeline milestones must be marked true before `/off` succeeds. The `/off` command is not an unconditional close — it is subject to the same `_check_phase_gates()` validation.

### Section 5: Test Changes (`tests/run-tests.sh`)

**Tests to update:**
- Remove timestamp assertions from token format checks (no `ts` field)
- Remove "expired token cleanup" test (no TTL)
- Remove "complete generates two tokens" test (no pre-generated "off")
- Simplify token validation to check `target` and `nonce` only

**New tests:**
- `/off` generates an "off" token with correct target
- Bare `set_phase review` in user message generates NO token (false-positive regression)
- Bare `set_autonomy_level auto` in user message generates NO token
- `.phase-tokens` in Bash command triggers deny from bash-write-guard (defense-in-depth)
- Token without matching target is rejected by `set_phase()`
- Token consumed after use — second `set_phase()` call with same target fails
- `/discuss some arguments here` still generates correct token (slash command with trailing text)

**Tests unchanged:**
- `run_with_auth` helper pattern
- `WF_SKIP_AUTH=1` bypass for non-BUG-3 tests
- Integration tests (hook → `set_phase()` flow)
- Forward-only auto-transition tests
- Autonomy token tests (format changes only — no `ts`)

## Non-Goals

- HMAC signing (deferred — see ADR rationale)
- CLAUDE_ENV_FILE integration (platform bugs make it unreliable)
- PreToolUse exit code 2 migration (unreliable per documented bugs)
- Changes to BUG-1 (autonomy mapping) or BUG-2 (echo chaining) fixes — both are solid
