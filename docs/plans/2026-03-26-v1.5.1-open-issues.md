# v1.5.1 Open Issues — Design Spec

> Addresses 7 open issues from ECC integration (v1.5.0–1.5.1). Grouped by complexity: housekeeping (C), packaging/DRY (B), architectural (A).

## Problem

After shipping the ECC integration (22 named agents, skill registry, 5 review agents), the v1.5.1 bugfix session identified 7 remaining issues ranging from trivial housekeeping to architectural gaps. These issues affect plugin reliability (agent dispatch failures), developer experience (redundant test runs), and maintainability (duplicated code, manual sync requirements).

## Approaches Considered

### Issue 1: Agent type registration

- **A (chosen): Inline agent prompts via Read + general-purpose dispatch.** Command files instruct Claude to read the agent `.md` file and embed its content into a `general-purpose` Agent call's prompt parameter. Agents remain as separate files for modularity.
- **B: Consolidate agents into command files.** Inline all agent instructions directly. Rejected: command files would balloon (complete.md from 380 to ~1000+ lines), agents used across phases (e.g., `review-verifier`) would be duplicated.
- **C: Document-only workaround.** Just document that `workflow-manager:*` should be dispatched as `general-purpose`. Rejected: already failed silently in last session.

### Issue 2: Redundant test execution

- **A (chosen): Git-based change gate.** Track `tests_last_passed_at` commit in workflow state. In COMPLETE, run `git diff --name-only <commit>..HEAD` — if only non-code files changed, present the diff as evidence and skip re-running.
- **B: Always re-run.** Accept redundancy. Rejected: violates feedback memory, wastes time.
- **C: Fork verification skill.** Override the external skill to allow cached results. Rejected: maintenance burden on external dependency.

### Issue 3: Plugin cache missing dirs

- **A (chosen): Update setup.sh cache sync.** Add `agents/` and `config/` to the items copied during version sync in Section B.

### Issue 4: Skill Resolution duplication

- **A (chosen): Extract to shared reference doc.** Create `plugin/docs/reference/skill-resolution.md`. Replace 5 inline blocks with a one-line reference.

### Issue 5: Duplicate plugin.json

- **A (chosen): Single canonical location.** Keep `.claude-plugin/plugin.json` as canonical. Remove `plugin/.claude-plugin/plugin.json`. Update `check-version-sync.sh` to check only 2 files (marketplace.json + canonical plugin.json).

### Issue 6: $STATE_FILE stale after setup_test_project

- **A (chosen): Re-derive STATE_FILE.** After `setup_test_project` creates a new `TEST_DIR`, re-export `STATE_FILE` pointing to the new directory's state file.

### Issue 7: Hardcoded model name in commit template

- **A (chosen): Dynamic model detection.** Replace the hardcoded `Claude Opus 4.6 (1M context)` with an instruction telling Claude to use its own model name from the system prompt environment section.

## Decision

- **Chosen approach:** All 7 issues addressed in a single implementation cycle, sequenced C → B → A (simple fixes first, architectural changes last).
- **Rationale:** Groups B and C are small enough that batching avoids workflow re-entry overhead. Group A benefits from the cleanup in B/C being done first.
- **Trade-offs accepted:** The agent dispatch pattern (Issue 1) adds one extra Read call per agent dispatch. This is a minor token cost vs. the alternative of inlining agents.
- **Risks identified:** The git-based change gate (Issue 2) could produce false negatives if a non-code file change (e.g., config) affects test behavior. Mitigated by limiting the "skip" to clearly non-code extensions.
- **Tech debt acknowledged:** Agent dispatch via Read + general-purpose is a workaround. If Claude Code adds custom agent type registration, we should migrate back to `workflow-manager:*` subagent_type.

#### C2: Fix $STATE_FILE after setup_test_project (Issue 6)

In `tests/run-tests.sh`, the `setup_test_project()` function (line 150-162) creates a new `TEST_DIR` via `mktemp -d` but doesn't update the `STATE_FILE` variable. The old `TEST_DIR` is `rm -rf`'d at line 151, leaving `$STATE_FILE` pointing to a deleted directory until the next `source workflow-state.sh` call re-derives it. While current tests always `source` before using `$STATE_FILE`, the gap is a latent bug for future test authors.

**Fix:** Add `STATE_FILE="$TEST_DIR/.claude/state/workflow.json"` at the end of `setup_test_project()`, after `export CLAUDE_PROJECT_DIR="$TEST_DIR"`.

**Also fix:** The `trap` at line 144 (`trap 'rm -rf "$TEST_DIR"' EXIT`) captures the initial `TEST_DIR` value. After `setup_test_project()` reassigns `TEST_DIR`, the trap holds a stale path. Move the trap into `setup_test_project()` so it always cleans up the current directory:

```bash
setup_test_project() {
    rm -rf "$TEST_DIR"
    TEST_DIR=$(mktemp -d)
    trap 'rm -rf "$TEST_DIR"' EXIT  # Update trap to current TEST_DIR
    # ... rest of function ...
    export CLAUDE_PROJECT_DIR="$TEST_DIR"
    STATE_FILE="$TEST_DIR/.claude/state/workflow.json"
}
```

### Group B: Packaging & DRY (Issues 3, 4)

#### B1: Plugin cache includes agents/ and config/ (Issue 3)

In `plugin/scripts/setup.sh` Section B (lines 83-100), the cache sync copies top-level items from old version dirs but doesn't explicitly ensure `agents/` and `config/` exist in the cache.

The real issue: when installed from the marketplace, Claude Code copies the entire `plugin/` directory as the plugin root. The `agents/` and `config/` directories ARE part of the plugin and SHOULD be copied by Claude Code's plugin system. The setup.sh cache sync (Section B) only handles version bumps within an already-cached plugin.

**Fix:** In the cache sync block, after copying from old version (line 92-98), also ensure source `agents/` and `config/` are in the new version cache:

```bash
# Ensure agents/ and config/ are in the cache (may be missing from older versions)
for dir in agents config; do
  if [ -d "$PLUGIN_ROOT/$dir" ] && [ ! -d "$CACHE_DIR/$SOURCE_VERSION/$dir" ]; then
    cp -r "$PLUGIN_ROOT/$dir" "$CACHE_DIR/$SOURCE_VERSION/$dir"
  fi
done
```

This is additive — it copies these dirs from the source plugin into the cache if they're missing, without disturbing existing cache content.

### Group A: Architectural (Issues 1, 2)

#### A1: Agent Dispatch Pattern (Issue 1)

**Problem:** The 28 agent `.md` files in `plugin/agents/` are specifications, but `workflow-manager:<name>` is not a registered `subagent_type` in Claude Code. Dispatching fails silently, falling back to generic behavior.

**Solution: Standardized dispatch pattern via shared reference doc.**

**Create** `plugin/docs/reference/agent-dispatch.md`:

```markdown
# Agent Dispatch Pattern

Claude Code does not support custom `subagent_type` values. The `workflow-manager:*`
agent types defined in `plugin/agents/` are specifications, not runtime-registered agents.

## How to dispatch a named agent

1. **Read** the agent file: `plugin/agents/<agent-name>.md`
2. **Extract** the agent's system prompt (everything after the YAML frontmatter closing `---`)
3. **Dispatch** using the Agent tool with:
   - `subagent_type`: `"general-purpose"`
   - `prompt`: Combine the agent's system prompt with the runtime context for this specific dispatch

## Example

To dispatch `plan-validator` with runtime context about the plan file:

1. Read `plugin/agents/plan-validator.md`
2. Use Agent tool:
   - `subagent_type: "general-purpose"`
   - `prompt: "<content of plan-validator.md after frontmatter>\n\n---\n\nRuntime context: Plan file is at docs/superpowers/specs/2026-03-26-example-design.md. Validate all deliverables."`

## Model selection

Agent files specify `model: inherit`. When dispatching as general-purpose, the parent
session's model is used automatically (this is the default behavior).

## Tools

Agent files list allowed tools in frontmatter (e.g., `tools: [Read, Grep, Glob]`).
When dispatching as general-purpose, all tools are available. The tool list in
frontmatter serves as documentation of what the agent needs, not as a restriction.
Review this section if Claude Code adds tool-scoping support for general-purpose agents.
```

**Update all 5 command files** to reference this dispatch pattern. For each command file:

1. Add a one-line reference near the top: `**Agent Dispatch:** Follow `plugin/docs/reference/agent-dispatch.md` — dispatch all agents as `general-purpose` with their `.md` file content as the prompt.`

#### A2: Git-based test change gate (Issue 2)

**Problem:** The `verification-before-completion` skill requires "fresh evidence in this message" before claiming tests pass. But tests already ran in IMPLEMENT phase and re-running in COMPLETE when no code changed is wasteful (violates feedback memory).

**Solution:** Add a `tests_last_passed_at` top-level field to workflow state (preserved across phase transitions) and use git diff as fresh evidence.

**Workflow state changes in `workflow-state.sh`:**

1. Add `set_tests_passed_at(commit_hash)` — stores commit hash as top-level field in workflow.json
2. Add `get_tests_passed_at()` — retrieves the stored commit hash
3. Add to `_read_preserved_state()`: `preserved_tests_passed=$(get_tests_passed_at)`
4. Add to the `set_phase()` preserved variables: declare `preserved_tests_passed`, include in jq state build
5. The field is preserved across ALL phase transitions (implement → review → complete) and cleared on `off` (cycle complete)

**Integration points in `set_phase()`:**
- Add `local preserved_tests_passed=""` to the local declarations (alongside line 371)
- Add `preserved_tests_passed=""` to the off-clearing block (alongside lines 381-385)
- Add `--arg tests_passed "$preserved_tests_passed"` to the jq arguments
- Add `+ (if $tests_passed != "" then {tests_last_passed_at: $tests_passed} else {} end)` to the jq conditional chain (alongside the existing `$obs_id` / `$tracked` / `$snapshot` conditionals)

**Workflow cmd changes in `workflow-cmd.sh`:**

Add `set_tests_passed_at|get_tests_passed_at` to the case statement dispatch.

**IMPLEMENT phase change** (`implement.md`):

After tests pass, store the commit. Insert immediately after the existing `set_implement_field "tests_passing" "true"` milestone marker:
```bash
.claude/hooks/workflow-cmd.sh set_tests_passed_at "$(git rev-parse HEAD)"
```

**Invariant:** Field is only set on test success, never on test failure. If tests never pass in IMPLEMENT, the field remains unset and COMPLETE falls through to running tests (safe default).

**COMPLETE phase change** (`complete.md`):

Before Step 1 (Plan Validation), add a test evidence gate:

```markdown
### Pre-validation: Test Evidence Gate

1. Read `tests_last_passed_at` from workflow state
2. If set, run: `git diff --name-only <tests_last_passed_at>..HEAD`
3. Classify changed files using a **safe-to-skip whitelist** (only skip if ALL changed files match):
   - **Safe to skip** (non-code): `docs/**/*.md`, `*.txt`, `.gitignore`, `LICENSE`, `README.md`, `CHANGELOG.md`
   - **Everything else is code** — including `*.sh`, `*.py`, `*.js`, `*.json`, `plugin/commands/*.md`, `plugin/agents/*.md`, `plugin/config/*`, `plugin/docs/reference/*.md`
   - Rule: if in doubt, treat as code (run tests). Only the whitelist above is safe.
4. If ALL changed files are in the safe-to-skip whitelist:
   - Present evidence: "No code files changed since tests passed at commit <hash>. Git diff shows only: <list of changed files>. Using previous test results as evidence."
   - Skip test re-run
5. If code files changed OR `tests_last_passed_at` is not set:
   - Run full test suite
   - Update `tests_last_passed_at` with current HEAD
```

This satisfies the verification skill's "fresh evidence" requirement — the `git diff` output IS fresh evidence proving the previous test results remain valid.

**Edge case:** If someone runs `/complete` without going through `/implement` (e.g., resuming a session), `tests_last_passed_at` won't be set. In this case, tests always run. This is the safe default.

**Tests:**
- Test `set_tests_passed_at` / `get_tests_passed_at` in workflow-state.sh
- Test that the field persists across phase transitions
- Test the classification logic (code vs non-code files)
