# Decision Record: Phase Token Security Model (BUG-3 Redesign)

**Date:** 2026-03-25
**Phase:** DISCUSS
**Origin:** BUG-3 — Claude can forge phase transition tokens during IMPLEMENT/REVIEW phases

## Problem

The phase token system (introduced in commit `3a4c292`) prevents Claude from self-triggering phase transitions by requiring one-time tokens generated by the `UserPromptSubmit` hook (which only fires on actual user input). However, the implementation has critical flaws:

1. **Token forgery during IMPLEMENT/REVIEW:** `bash-write-guard.sh` allows all writes in these phases (`exit 0` at line 122-124), so Claude can `mkdir -p .phase-tokens && echo '{"target":"off",...}' > .phase-tokens/fake` to create valid tokens.
2. **Bare command matching:** Lines 34-35 and 48-49 of `user-phase-token.sh` match `set_phase` and `set_autonomy_level` as substrings in any user message (e.g., "why did set_phase off fail?" generates a token).
3. **Unnecessary TTL:** The 60-second expiry adds complexity and actively breaks long-running pipelines (the `/complete` pipeline takes minutes, so the "off" token expires before Step 9).
4. **Pre-generated "off" token for `/complete`:** Conflates user-initiated workflow close with automated pipeline completion. OFF transitions should always be user-initiated.
5. **Future-timestamp bypass:** `$((now - ts))` with `ts > now` produces a negative that passes `< 60`. Patched in `b2b3177` but the TTL system is unnecessary.

## Research: Claude Code Hook Platform Constraints

Extensive research (March 2026) into the Claude Code hook system revealed significant platform limitations that constrain the solution space.

### CLAUDE_ENV_FILE

- **Only available in SessionStart hooks.** Direct quote from docs: "`CLAUDE_ENV_FILE` is available for SessionStart hooks. Other hook types do not have access to this variable."
- Variables written to CLAUDE_ENV_FILE are sourced before **Bash tool calls** only, not before other hook invocations.
- Known bugs: empty for plugin hooks (fixed Dec 2025, [#11649](https://github.com/anthropics/claude-code/issues/11649)), empty in some versions ([#15840](https://github.com/anthropics/claude-code/issues/15840)), not sourced on resumed sessions ([#12418](https://github.com/anthropics/claude-code/issues/12418)).

**Impact:** An HMAC secret stored via CLAUDE_ENV_FILE cannot be read by UserPromptSubmit hooks for token signing. This breaks the original HMAC design's Layer 2.

### PreToolUse Hook Blocking

Both documented blocking mechanisms are unreliable:

**Exit code 2:**
- Documented as blocking, but fails in practice for Bash ([#23284](https://github.com/anthropics/claude-code/issues/23284)), Write/Edit ([#13744](https://github.com/anthropics/claude-code/issues/13744)), Task ([#26923](https://github.com/anthropics/claude-code/issues/26923) — 19/19 blocks failed).
- In bypass mode, commands execute before hooks finish ([#20946](https://github.com/anthropics/claude-code/issues/20946)).

**`permissionDecision: "deny"`:**
- Closed as NOT_PLANNED ([#4669](https://github.com/anthropics/claude-code/issues/4669)).
- Completely bypassed when tool is in allow list ([#18312](https://github.com/anthropics/claude-code/issues/18312), [#13214](https://github.com/anthropics/claude-code/issues/13214)).

**Impact:** PreToolUse hooks cannot be relied upon as a security boundary. They are useful as defense-in-depth (raising the bar) but not as a primary enforcement mechanism.

### Inter-Hook Communication

- No built-in mechanism for hooks to share state ([#18654](https://github.com/anthropics/claude-code/issues/18654), closed NOT_PLANNED).
- Documented approach: write to files in `$CLAUDE_PROJECT_DIR` or `$CLAUDE_PLUGIN_DATA`.
- `$CLAUDE_PLUGIN_DATA` is available to ALL hook types (SessionStart, UserPromptSubmit, PreToolUse, etc.).
- `session_id` is available via stdin JSON in all hooks but is not a secret (visible to Claude).

### File Permissions

- `chmod` is ineffective — Claude's Bash tool runs as the same UID as hooks.
- File-based access control cannot prevent same-user reads.

## Approaches Considered

### Approach A: HMAC-Signed Tokens via `$CLAUDE_PLUGIN_DATA`

Store HMAC secret in `$CLAUDE_PLUGIN_DATA/.hmac-secret` (accessible to all hooks). UserPromptSubmit signs tokens with HMAC-SHA256. `set_phase()` verifies signatures. PreToolUse blocks secret access (defense-in-depth).

- **Pros:** Cryptographic verification, defense-in-depth layers.
- **Cons:** Secret is readable by Claude (same UID, unreliable PreToolUse blocking). If Claude reads the secret, the HMAC adds no value — the real security is still UserPromptSubmit. More complex implementation for marginal security gain.

### Approach B: Simplified File-Token with Application-Level Enforcement (Chosen)

Trust the UserPromptSubmit platform boundary. Generate unsigned-but-unique tokens. Protect via application-level checks in `set_phase()` plus defense-in-depth PreToolUse guards.

- **Pros:** Simplest approach, minimal code changes, same practical security as cc-sessions (battle-tested). Easy to audit and maintain.
- **Cons:** If Claude bypasses PreToolUse AND writes to `.phase-tokens/` during IMPLEMENT/REVIEW, it can forge tokens. No cryptographic verification.

### Approach C: HMAC + Session-Scoped Secret via Double-Source

Write secret to both CLAUDE_ENV_FILE (for Bash) and `$CLAUDE_PLUGIN_DATA` (for hooks). Strongest defense-in-depth but most complex, with marginal practical security gain.

- **Pros:** 3 independent layers must fail for forgery. Per-session rotation.
- **Cons:** Most complex. Secret still readable by Claude. Marginal gain over Approach B.

### Prior Art: cc-sessions (GWUDCAP/cc-sessions)

[cc-sessions](https://github.com/GWUDCAP/cc-sessions) solves the same problem with no cryptographic tokens at all. Their model: UserPromptSubmit is the trust boundary, PreToolUse guards the state file, behavioral anchoring reinforces the boundary. They accept the risk that PreToolUse blocking is unreliable and rely on the platform guarantee that Claude cannot trigger UserPromptSubmit.

## Decision

**Chosen approach:** Approach B — Simplified File-Token with Application-Level Enforcement.

### Rationale

The security of ALL three approaches ultimately rests on the same platform guarantee: **UserPromptSubmit only fires on actual user input.** Claude cannot trigger it. This is the real security boundary.

HMAC (Approaches A and C) adds cryptographic verification, but the secret is stored in `$CLAUDE_PLUGIN_DATA` which Claude can read via Bash (same UID). PreToolUse guards that would block secret access are unreliable per documented bugs. Therefore, HMAC adds implementation complexity without meaningful security gain — if Claude can read the secret and write files, it can forge a valid HMAC token.

The practical attack requires Claude to:
1. Know the token file format (discoverable — code is open source)
2. Know the token directory path (discoverable)
3. Be motivated to forge (requires prompt injection or adversarial instruction)
4. Be in IMPLEMENT or REVIEW phase (the only phases with write access)

This is the same risk that cc-sessions accepts, and it works in production. The risk is bounded: it requires adversarial intent, and the blast radius is limited to phase transitions (not data loss or code execution).

### Trade-offs Accepted

- **Token forgery during IMPLEMENT/REVIEW is possible** if Claude is adversarially instructed. This is accepted because: (a) the platform guarantee (UserPromptSubmit) is the real boundary, (b) PreToolUse defense-in-depth raises the bar, (c) the attack requires active adversarial intent, not accidental behavior.
- **No cryptographic verification.** Simpler code, easier to audit, same practical security.

### Risks Identified

- If Anthropic introduces a bug where UserPromptSubmit fires on non-user input, the entire model collapses. This would also break cc-sessions and every other hook-based enforcement system.
- Prompt injection that instructs Claude to forge tokens during IMPLEMENT phase is the most realistic attack vector.

### Tech Debt Acknowledged

- PreToolUse blocking should be upgraded if/when Anthropic fixes the blocking bugs. The current `emit_deny` (JSON stdout) approach should be replaced with reliable blocking once available.
- HMAC signing could be added later as a defense-in-depth layer if the threat model changes.

## Changes from Current Implementation

| Change | Why |
|--------|-----|
| Remove bare `set_phase`/`set_autonomy_level` matching | Prevents false-positive token generation from natural conversation |
| Remove TTL/expiry system | Unnecessary complexity, breaks long pipelines, future-timestamp vulnerability |
| Remove pre-generated "off" token for `/complete` | OFF is always user-initiated |
| Add `/off` command | User must explicitly close the workflow |
| Add PreToolUse guard for `.phase-tokens/` writes | Defense-in-depth (unreliable but raises the bar) |
| Keep atomic mv-then-validate consumption | Prevents TOCTOU races |
| Keep `WF_SKIP_AUTH=1` test bypass | Required for existing test infrastructure |
| Keep forward-only auto-transition | Correct autonomy=auto behavior |
