# v1.12.0 Design: Robustness Fixes, Tooling Extraction, COMPLETE Pipeline Refactor

**Date:** 2026-03-27
**Version:** v1.12.0
**Issues:** #4488, #4570, #4581, #4586, #4590

## Problem

Five open issues spanning security false positives, tech debt, missing UX polish, personal tooling mixed into the plugin, and a structural enforcement gap in the COMPLETE pipeline:

1. **#4586 — FILE_OPS false positives:** `bash-write-guard.sh` FILE_OPS regex matches command names mid-word (e.g., `rm` inside `iTerm`), causing false blocks during legitimate COMPLETE phase operations like `gh issue edit`.

2. **#4570 — Three tech debt items:**
   - Worktree cleanup in COMPLETE Step 6 lacks merge-verification — can delete unmerged branches if push was deferred.
   - ~67 stale references to removed `save_completion_snapshot` across 5 historical design docs mislead future readers.
   - Unused `.claude/commands/` entry in `COMPLETE_WRITE_WHITELIST` — dead code.

3. **#4590 — Command output not reliably presented:** WFM command files use `!` backtick syntax but don't explicitly instruct Claude to present the output, unlike the cc-sessions pattern. Claude may silently consume the output.

4. **#4581 — YubiKey/iTerm tooling mixed into plugin repo:** Personal tooling (`tools/yubikey-setup/`, `tools/iterm-launcher/`) is committed to the WFM repo. These are user-specific, not plugin functionality.

5. **#4488 — COMPLETE Step 7 skipping:** Claude skips Steps 4-7 during COMPLETE, jumping from validation (Step 3) directly to commit (Step 5) or handover (Step 8). This has occurred three times in one session. Existing Check 9 coaching fires on specific tool patterns (git commit, save_observation) but can't detect step narration without execution.

## Approaches Considered

### FILE_OPS Fix (#4586)

**A: Word boundary `\b`** — `\brm[[:space:]]`. Clean but `\b` is PCRE/GNU, not portable ERE.

**B: Start-of-command anchor** — `(^|[;&|[:space:]])rm[[:space:]]`. Portable ERE, precise. More verbose.

**C: Accept false positives** — Guard is "a speed bump, not a wall." Live with the friction.

**Chosen: B.** Portable, precise, and the false positive on `gh issue edit` with "iTerm" in the body is a real usability problem.

### COMPLETE Step 7 Skipping (#4488)

**A: More milestone gates** — Add `tech_debt_audited`, `issues_created` fields. Extend Check 9 to verify before allowing commit. Same pattern as existing enforcement.

**B: Explicit step checklist** — Numbered checklist Claude must tick off. Visible but still relies on Claude following instructions.

**C: Structural pipeline** — Break Steps 4-8 into discrete agent dispatches. Each agent handles steps and sets milestones. Orchestrator can't skip because each step is a mandatory agent call with a required result.

**Chosen: C.** Options A and B add speed bumps to a problem that keeps recurring despite existing speed bumps. Check 9 was already supposed to catch this — it didn't. The review pipeline (5 parallel agents) proves the structural pattern works. Three sequential agents for Steps 4-8 provide real enforcement.

### Tooling Extraction (#4581)

**A: New separate repo** — Clean separation, version-controlled independently.

**B: Gitignored local overlay** — Co-located but not committed. Zero friction, no version control.

**C: Conditional blocks** — Guard behind env var. Least disruptive but adds conditionals.

**Chosen: A.** Separate repo at `azevedo-home-lab/claude-code-tools` (private). Files are moved (not just deleted), install URLs updated.

## Decision

- **Bundling strategy:** Approach C — single feature branch, stacked commits. One release cycle, granular git history for bisect.
- **Commit order:** Dependencies flow downward. Regex fix first (standalone), then tech debt, then command files, then extraction, then pipeline refactor, then docs-detector.

## Detailed Design

### Commit 1: FILE_OPS Word Boundary Fix (#4586)

**File:** `plugin/scripts/bash-write-guard.sh` line 26

Current:
```bash
FILE_OPS='(cp[[:space:]]|mv[[:space:]]|rm[[:space:]]|install[[:space:]]|patch[[:space:]]|ln[[:space:]]|touch[[:space:]]|truncate[[:space:]])'
```

Proposed:
```bash
FILE_OPS='((^|[;&|[:space:]])(cp|mv|rm|install|patch|ln|touch|truncate)[[:space:]])'
```

Each command must be preceded by line start, shell operator, or whitespace. Prevents matching inside words like `iTerm`, `perform`, `transform`.

**Tests to add** in `tests/run-tests.sh`:
- `gh issue edit` with "iTerm tooling" in body → ALLOW
- `rm file.txt` → DENY
- `echo foo && rm file.txt` → DENY
- `perform cleanup` → ALLOW
- `transform data` → ALLOW

### Commit 7: COMPLETE Pipeline Agent Refactor (#4488)

Replace COMPLETE Steps 4-8 (currently inline instructions) with 3 sequential agent dispatches.

**New agent files:**

#### `plugin/agents/commit-push-agent.md`
**Scope:** Steps 4-6 (docs detection, commit/push, branch integration/worktree cleanup)
**Inputs:** Decision record path, changed files list, autonomy level
**Outputs:** Commit hash, push status, branch integration status
**Milestones set:** `docs_checked`, `committed`, `pushed`
**Review gates included:** docs-reviewer, commit-reviewer (dispatched within the agent)
**User interaction:** Push confirmation routed through structured response — agent returns `PUSH_PENDING` if confirmation needed, orchestrator relays to user and feeds answer back.

#### `plugin/agents/tech-debt-agent.md`
**Scope:** Step 7 (review tracked observations, collect/categorize findings, save observations, create GitHub issues, temp file cleanup)
**Inputs:** Decision record path, tracked observations list, validation findings from Steps 1-3
**Outputs:** Category observations saved, GitHub issues created, KEEP_IDS list, new observation IDs
**Milestones set:** `tech_debt_audited`
**Review gates included:** tech-debt-reviewer (dispatched within the agent)
**Shared state:** Writes KEEP_IDS and new observation IDs to `.claude/tmp/tech-debt-handoff.json` for the handover agent.

#### `plugin/agents/handover-agent.md`
**Scope:** Step 8 (write handover observation, update tracked observations list)
**Inputs:** Decision record path, `.claude/tmp/tech-debt-handoff.json` (from tech-debt-agent)
**Outputs:** Handover observation ID
**Milestones set:** `handover_saved`
**Review gates included:** handover-reviewer (dispatched within the agent)
**Wraps existing agent:** This agent orchestrates the existing `handover-writer.md` agent (which does the actual writing) and the `handover-reviewer.md` agent (review gate). It does NOT replace them — it adds the tracked observations update and milestone setting around them.
**Atomic update:** Builds final tracked observations list (KEEP_IDS + handover ID + new tech debt IDs) in a single `set_tracked_observations` call.

## Trade-offs Accepted

1. **More tokens per COMPLETE run** (~30-40% more for Steps 4-8) due to 3 agent context windows instead of inline execution. This is the cost of structural enforcement over behavioral enforcement.
2. **Tooling gap during extraction** — between deletion from this repo and new repo setup, install URLs will 404. Acceptable because user controls timing and the new repo is created first.
3. **Stale reference detection is scoped** — only catches references near changed files, not repo-wide. Full sweeps deferred to the cleanup agent concept.
4. **FILE_OPS regex is more verbose** — harder to read at a glance, but the variables are already documented and the pattern is correct.

## Tech Debt Acknowledged

- Historical docs still contain 150+ snapshot references (now with deprecation banners, not rewritten)
- Check 9 coaching in post-tool-navigator.sh may become partially redundant after agent pipeline — keep as defense-in-depth for now, evaluate removal in future version
- The commit-push-agent handles user interaction (push confirmation) via structured response routing — this is a new pattern that may need refinement based on real usage
