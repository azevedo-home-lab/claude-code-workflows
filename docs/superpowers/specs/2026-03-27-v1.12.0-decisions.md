## Problem

Five open issues requiring a combined release:
- #4586: FILE_OPS regex false positives (mid-word matching)
- #4570: Three tech debt items (merge-verification, stale refs, unused whitelist)
- #4590: Command output not reliably presented to user
- #4581: YubiKey/iTerm personal tooling mixed into plugin repo
- #4488: COMPLETE Step 7 repeatedly skipped due to behavioral enforcement limits

## Approaches Considered (DISCUSS phase — diverge)

### FILE_OPS Fix
- **A: `\b` word boundary** — Clean but non-portable ERE
- **B: Start-of-command anchor** — `(^|[;&|[:space:]])cmd[[:space:]]` — portable, precise
- **C: Accept false positives** — Live with friction

### COMPLETE Pipeline
- **A: More milestone gates** — Same class of enforcement already failing
- **B: Explicit checklist** — Still relies on Claude following instructions
- **C: Structural agent pipeline** — Discrete dispatches, can't skip

### Tooling Extraction
- **A: Separate repo** — Clean separation, version controlled
- **B: Gitignored overlay** — Co-located, no version control
- **C: Conditional blocks** — Env var gating, adds complexity

### Bundling Strategy
- **A: Single release** — One batch, larger blast radius
- **B: Two micro-releases** — Smaller chunks, two cycles
- **C: Feature branch with stacked commits** — One cycle, granular history

## Decision (DISCUSS phase — converge)

- **FILE_OPS:** Approach B — start-of-command anchor. Portable and precise.
- **COMPLETE Pipeline:** Approach C — structural agent pipeline. Three sequential agents replace inline Steps 4-8.
- **Tooling Extraction:** Approach A — separate repo at `azevedo-home-lab/claude-code-tools` (private).
- **Bundling:** Approach C — feature branch, stacked commits, one release.

**Rationale:**
- FILE_OPS: The false positive on `gh issue edit` with "iTerm" blocked real work. Portable ERE is correct even if more verbose.
- COMPLETE: Check 9 coaching was already supposed to prevent step skipping — it didn't. Adding more milestone fields treats the symptom. The review pipeline proves agent-based structural enforcement works.
- Tooling: User-requested. Separate repo gives version control and clean separation.
- Bundling: Single cycle avoids overhead. Stacked commits give rollback granularity.

**Trade-offs accepted:**
- ~30-40% more tokens per COMPLETE run (3 agent windows vs inline)
- Brief tooling URL gap during extraction (user controls timing)
- Stale reference detection scoped to changed files only (full sweep deferred)
- Historical docs get deprecation banners, not content rewrites

**Risks identified:**
- commit-push-agent user interaction routing (push confirmation) is a new pattern
- Agent pipeline adds latency to COMPLETE phase
- FILE_OPS regex change could miss edge cases in command parsing

**Constraints applied:**
- ERE portability requirement (macOS grep -E)
- Sequential agent execution (Steps 4-8 have data dependencies)
- Historical docs preserved as design records

**Tech debt acknowledged:**
- Check 9 coaching partially redundant after agent pipeline (keep as defense-in-depth)
- 150+ snapshot references now bannered, not rewritten

**Additional enhancement from discussion:**
- Docs-detector agent gets stale reference detection capability
- This was user feedback: deprecation should be a regular concern for the documentation agent
